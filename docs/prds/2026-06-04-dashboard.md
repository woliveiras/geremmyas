# PRD: Dashboard

**Date:** 2026-06-04
**Author:** William Oliveira / AI-assisted
**Status:** Draft

## Problem

Geremmyas users follow a spec-driven development workflow that produces
structured markdown artifacts: specs with frontmatter (`spec.md`, `plan.md`,
`tasks.md`), PRDs, and bugfix documents. The `specs/README.md` index serves as
the single overview of all specs across families and phases.

This approach works well for small projects (10–20 specs). It breaks down at
scale:

1. **Unreadable index.** Projects with 100+ specs produce a README.md that is
   too long to scan visually. Families, phases, and dependencies get lost in a
   wall of markdown tables.
2. **No metrics.** Velocity, lead time (created → implemented), review time
   (created → approved), and implementation time (approved → implemented) are
   not tracked. Teams cannot answer "are we shipping faster?" or "which family
   is blocked?"
3. **Non-technical stakeholders cannot consume markdown.** PMs, POs, and
   stakeholders need a visual board and progress overview — not raw `.md` files
   on GitHub.
4. **Agent context cost.** Agents reading a 200+ line index pay unnecessary
   token cost when they only need specs from one family or status.

The problem exists across 4+ projects that use geremmyas conventions, with the
largest project already approaching 100 specs organized in multiple families and
phases.

## Users

| Persona | Need | Current workaround |
| --- | --- | --- |
| **Developer (solo or team)** | Quick status overview, navigate specs | Read `specs/README.md` in editor |
| **PM / PO / Stakeholder** | Board view, progress metrics, search | Ask developer or read markdown on GitHub |
| **Coding agent** | Filtered spec list, specific spec content | Read full `specs/README.md` + individual files |

## Goals

1. Provide a visual, navigable HTML dashboard generated from existing spec
   artifacts — zero workflow change for developers and agents.
2. Extract delivery metrics from git history without requiring manual timestamp
   entry.
3. Ship as a built-in geremmyas CLI command so every geremmyas user gets it.
4. Keep the `.md` files as source of truth — the dashboard is a read-only
   rendered view.

## Non-goals

- Replacing `.md` files with a database.
- Multi-repo unified dashboard (each repo runs its own instance).
- Real-time collaboration or editing through the UI.
- Authentication or deployment to a remote server.
- Search functionality (future scope).
- `geremmyas list` CLI query command (future scope — agents use the generated
  README for now).
- Including deprecated specs in metrics calculations (future scope).

## Solution overview

A new `geremmyas dashboard` command that:

1. **Scans** `specs/`, `docs/prds/`, `docs/bugfixes/` for markdown files with
   YAML frontmatter.
2. **Parses** spec metadata (number, title, family, phase, status, depends_on,
   origin, owner) and task progress (checkbox counts from `tasks.md`).
3. **Extracts dates from git history** using batch `git log` commands: creation
   date, approval date, implementation date for each spec.
4. **Caches** git history results in `.geremmyas-cache/gitdates.json` for fast
   incremental updates.
5. **Generates** a static HTML site with embedded CSS/JS (no external
   dependencies at runtime).
6. **Auto-generates a simplified `specs/README.md`** as a compact agent-readable
   index (one line per spec: number, title, status). This replaces the
   hand-maintained index tables.
7. **Optionally serves** the site on localhost with `--serve` and watches for
   file changes with `--watch`.

All templates and static assets are embedded in the geremmyas binary via
`go:embed`. No build step, no Node.js, no npm.

### Auto-generated `specs/README.md`

Every run of `geremmyas dashboard` overwrites `specs/README.md` with a compact,
agent-optimized index. Format: one line per spec, grouped by family. No
elaborate tables — just enough for an agent to locate specs by status without
reading every folder.

Example output:

```markdown
# Specs Index (auto-generated by geremmyas dashboard)

## Onboarding de Clínicas (0001–0049)

- 0001 RLS school pillar com clinic_id — Implemented
- 0002 Validação de clinic_id no merge — Implemented
- 0010 Role ADMIN global — Implemented
- 0032 Wizard de criação de clínica — Approved
- 0035 Importação de dados legados — In Review

## Platform (0100–0199)

- 0100 Dashboard command — Draft
```

The `sdd` pack template for `specs/README.md` will include a header comment
marking the file as auto-generated so agents know not to hand-edit it:

```markdown
<!-- geremmyas:generated — do not edit manually, regenerated by `geremmyas dashboard` -->
```

## Views

### 1. Overview (index)

Summary cards per family showing: total specs, specs by status (color-coded),
active phase, overall progress percentage. Click a family card to drill into
it.

Global summary bar at top: total specs, implemented count, in-progress count,
pending count.

### 2. Board

One kanban board **per family** (not a single global board). Each family page
includes its own board with columns: **Draft → In Review → Approved →
Implemented**. Each card shows spec number, title, phase badge, and task
progress bar. Filter by phase via dropdown within the family.

The family detail page (from view 5) IS the board — no separate board page.

Deprecated specs are hidden by default (toggle to show).

### 3. Spec detail

Full rendered spec content (markdown → HTML). Sidebar with:
- Status badge
- Family / phase
- Owner
- Dependencies (linked)
- Origin PRD (linked)
- Task progress (from `tasks.md` checkboxes)
- Timeline: created → approved → implemented dates (from git)

Links to `plan.md` and `tasks.md` rendered pages.

### 4. Metrics

Charts (Chart.js):

- **Velocity:** specs reaching "Implemented" per week/month (bar chart).
- **Lead time trend:** median time from created → implemented over time (line
  chart).
- **Phase breakdown:** specs by status per phase per family (stacked bar).
- **Family progress:** horizontal progress bars per family.

Summary cards:
- Average lead time (created → implemented)
- Average review time (created → approved)
- Average implementation time (approved → implemented)
- Specs in progress right now

### 5. Family detail

Specs grouped by phase within a family. Phase progress bars. Dependency
arrows or list showing blocked/blocking relationships.

## CLI interface

```
geremmyas dashboard [flags]

Flags:
  --output DIR    Output directory (default: .geremmyas/dashboard)
  --serve         Start local HTTP server after generating
  --port PORT     Server port (default: 8080)
  --watch         Rebuild on spec file changes (implies --serve)
  --no-git        Skip git history extraction (faster, no date metrics)
  --no-cache      Force full git rescan (ignore cache)
```

## Technical approach

### Parsing

Reuse and extend existing `internal/cli/frontmatter.go` for YAML frontmatter
extraction. The `sdd` pack template for `specs/README.md` will be updated to
document required frontmatter fields, ensuring specs created in new projects
are dashboard-compatible from the start. New parser scans:

- `specs/NNNN-*/spec.md` — frontmatter + body
- `specs/NNNN-*/plan.md` — existence check + body
- `specs/NNNN-*/tasks.md` — checkbox regex: `- \[ \]`, `- \[~\]`, `- \[x\]`
- `docs/prds/*.md` — frontmatter (title, date)
- `docs/bugfixes/*.md` — frontmatter or filename date

Family and phase grouping derived from spec frontmatter (`family`, `phase`
fields).

### Git dates

Batch extraction with two commands:

```bash
# Creation dates (one call for all specs)
git log --all --name-only --format="COMMIT:%H %aI" -- "specs/*/spec.md"

# Status change dates (one call, parse diffs)
git log --all -p --format="COMMIT:%H %aI" -- "specs/*/spec.md"
# Filter for +status: / -status: lines to detect transitions
```

Results cached in `.geremmyas-cache/gitdates.json` with `last_commit` SHA.
Incremental update: `git log <last_commit>..HEAD -- "specs/*/spec.md"`.

Expected performance:
- Cold (no cache, 100 specs): 2–5 seconds
- Warm (cached, incremental): < 500ms
- Watch mode: instantaneous (cache + file events)

### HTML generation

Go `html/template` with embedded template files. Static assets (CSS, Chart.js,
minimal JS) vendored and embedded via `go:embed`.

No framework. Semantic HTML + CSS Grid/Flexbox for layout. Dark/light theme
via `prefers-color-scheme`. Responsive for screen sharing in dailies.

### Serve mode

`net/http` from stdlib. File watcher via `fsnotify` (or polling fallback).
On change: re-parse affected files, regenerate HTML, signal browser reload
via Server-Sent Events or WebSocket.

### Cache

`.geremmyas-cache/` directory added to default `.gitignore` recommendation.
Cache file is JSON — human-readable, debuggable.

### Embedding

New embed directive in `assets.go` or separate `dashboard_assets.go`:

```go
//go:embed dashboard_assets/**
var DashboardAssets embed.FS
```

## Data model

| Entity | Source | Key fields |
| --- | --- | --- |
| Spec | `specs/NNNN-*/spec.md` frontmatter | number, slug, title, family, phase, status, depends_on, origin, owner |
| Plan | `specs/NNNN-*/plan.md` | exists (bool), content |
| Tasks | `specs/NNNN-*/tasks.md` | total, done, in_progress, pending |
| PRD | `docs/prds/*.md` | title, date, slug |
| Bugfix | `docs/bugfixes/*.md` | title, date, slug |
| SpecDates | git history | created_at, approved_at, implemented_at |
| Metrics | computed | velocity, avg_lead_time, avg_review_time, avg_impl_time |

## Implementation order

Specs are ordered by dependency, not by release. All specs ship together in a
single release. The order below defines the build sequence for development:

1. **0001** — Parser + data model (foundation for everything)
2. **0002** — Overview HTML + auto-generated README + sdd template update
3. **0003** — Board view (per-family kanban, extends family page from 0002)
4. **0004** — Git dates + cache + metrics charts
5. **0005** — Serve + watch mode
6. **0006** — PRD + bugfix linking + dependency visualization

All six specs must be implemented before release.

## Success criteria

1. Running `geremmyas dashboard` in a project with 100 specs produces a
   complete HTML dashboard in under 10 seconds (cold) or under 2 seconds
   (cached).
2. PM/PO can open the generated `index.html` and understand project status
   without reading any `.md` file.
3. Metrics page shows velocity and lead time derived from git history without
   any manual annotation.
4. Developer workflow is unchanged — agents and humans keep writing `.md` files
   as before.
5. Dashboard runs with zero external dependencies (no Node.js, no Docker, no
   database).

## Risks

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Git history unreliable after rebase/squash | Dates are approximate | Document that dates are best-effort; offer frontmatter fallback |
| Specs without frontmatter | Parser cannot group by family/phase | Graceful fallback: "Ungrouped" family; warn in output |
| Large repos slow git log | Cold generation > 10s | Cache makes repeated use fast; `--no-git` skips entirely |
| Chart.js size in embedded binary | ~200KB added to binary | Acceptable for a CLI tool; minified version |
| Template complexity grows | Maintenance burden | Keep views minimal; no SPA, no React — server-rendered HTML |

## Decisions (resolved)

1. **Auto-generate `specs/README.md`?** Yes. Compact one-line-per-spec format
   optimized for agent reading. Replaces the hand-maintained index.
2. **Update `sdd` pack template with frontmatter instructions?** Yes. Included
   in v0.1 so new projects are dashboard-compatible from day one.
3. **Build `geremmyas list` CLI query?** Not now. Agents use the auto-generated
   README. Revisit if agent workflows need structured output.
4. **Deprecated specs in metrics?** No. Excluded from velocity/lead-time
   calculations. Revisit if needed.
